## 1. Загальна архітектура

Система побудована у вигляді **мікросервісної архітектури**.

Кожен сервіс є незалежно розгортуваним компонентом, що взаємодіє з іншими через REST API.

```
[Client App]
     │
     ▼
[API Gateway]
     │
  ┌──┼──────────────────┐
  │  │                  │
  ▼  ▼                  ▼
[User   [Booking     [Search
Service] Service]    Service]
           │
           ▼
      [Payment
       Service]
```

---

## 2. Основні компоненти

| Компонент | Відповідальність |
|-----------|-----------------|
| **API Gateway** | Єдина точка входу, маршрутизація запитів |
| **User Service** | Реєстрація, авторизація, профіль |
| **Booking Service** | Створення, скасування, перегляд бронювань |
| **Payment Service** | Обробка платежів через зовнішній шлюз |
| **Search Service** | Пошук квитків за параметрами |

---

## 3. API Ендпоінти

### Користувачі
```
POST   /api/auth/register    — реєстрація
POST   /api/auth/login       — вхід (отримання JWT)
GET    /api/users/me         — профіль поточного користувача
```

### Квитки та бронювання
```
GET    /api/tickets          — пошук квитків
POST   /api/bookings         — створити бронювання
GET    /api/bookings         — список бронювань користувача
DELETE /api/bookings/{id}    — скасувати бронювання
```

### Платежі
```
POST   /api/payments         — ініціювати оплату
GET    /api/payments/{id}    — статус оплати
```

## 5. Безпека

- Всі запити захищені HTTPS
- Автентифікація через **JWT Bearer Token**
- Термін дії токену: 24 години
- Платіжні дані не зберігаються в системі — передаються напряму до шлюзу

---

## 6. Взаємодія між сервісами

- Сервіси взаємодіють через **внутрішні HTTP-запити**
- Для асинхронних операцій (наприклад, підтвердження платежу) планується використання черг повідомлень

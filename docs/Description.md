# Лабораторна робота №7

## Проектування архітектури програмної документації та системи забезпечення якості

## Тема

Проектування архітектури програмної документації та системи забезпечення якості

## Мета роботи

Метою лабораторної роботи є:

* формування системного уявлення про роль документації у життєвому циклі програмного забезпечення;
* набуття навичок класифікації та проектування програмної документації;
* ознайомлення з архітектурними документами SSD, SDD, ISD;
* встановлення зв’язку між вимогами, архітектурою та тестуванням.

---

# 1. Аналіз системи та цільової аудиторії

## 1.1 Опис системи

REST API сервіс для бронювання квитків призначений для обробки запитів клієнтів щодо пошуку, бронювання та оплати квитків. Система надає можливість створення замовлень, перегляду доступних рейсів або подій, а також керування користувачами.

Основними користувачами є клієнтські додатки (web або mobile), які взаємодіють із сервером через HTTP-запити. Система підтримує роботу з платежами та має інтеграцію з платіжними шлюзами.

Сервіс побудований у вигляді REST API, що дозволяє масштабувати його та інтегрувати з іншими системами.

Формат взаємодії — виключно API (JSON через HTTP).

## 1.2 Цільова аудиторія документації

* **Розробники (Backend / Frontend)** — для розуміння API та архітектури
* **Тестувальники (QA)** — для створення тест-кейсів
* **DevOps інженери** — для розгортання та підтримки
* **Бізнес / замовник** — для розуміння функціональності системи

# 3. SSD (System Specification Document)

## 3.1 Призначення системи

Система призначена для надання API для бронювання квитків, включаючи пошук, створення, оплату та скасування замовлень.

## 3.2 Основні функції

* Реєстрація та автентифікація користувачів
* Пошук доступних квитків
* Створення бронювання
* Оплата бронювання
* Скасування замовлення
* Перегляд історії замовлень

## 3.3 Обмеження та припущення

* Система працює через REST API
* Використовується одна основна база даних
* Зовнішній платіжний сервіс
* Очікується високе навантаження

## 3.4 Нефункціональні вимоги

* Продуктивність: не менше 500 запитів/хв
* Час відповіді: до 300 мс
* Доступність: 99.9% uptime
* Безпека: HTTPS, JWT авторизація
* Масштабованість: горизонтальне масштабування

# 4. SDD (Software Design Document)

## 4.1 Загальна архітектура

Система побудована у вигляді **мікросервісної архітектури**.

## 4.2 Основні компоненти

* **API Gateway** — точка входу
* **User Service** — керування користувачами
* **Booking Service** — бронювання
* **Payment Service** — обробка платежів
* **Search Service** — пошук квитків

## 4.3 Взаємодія між компонентами

Сервіси взаємодіють через REST API та внутрішні HTTP-запити.

### Приклад API

```
POST /api/bookings
GET /api/tickets
POST /api/payments
```

# 7. Контрольні питання

## 1. Чим SSD відрізняється від SDD?

SSD описує **що система повинна робити (вимоги)**, тоді як SDD описує **як система реалізована (архітектура)**.

## 2. Яка роль ISD у життєвому циклі ПЗ?

ISD визначає, як система розгортається, масштабується та підтримується в реальному середовищі.

## 3. Навіщо потрібна матриця простежуваності?

Вона дозволяє зв’язати вимоги з тестами, щоб гарантувати повне покриття функціоналу.

## 4. Чому документація впливає на якість продукту?

Документація забезпечує розуміння системи всіма учасниками процесу, зменшує кількість помилок та покращує підтримку продукту.

1. У чому полягає принцип Documentation as Code?
Documentation as Code (DaC) — це підхід, за якого документація ведеться так само, як і програмний код: зберігається у Git-репозиторії, пишеться у текстовому форматі (Markdown), проходить code review через Pull Request, і автоматично публікується через CI/CD pipeline. Це означає, що документація є частиною репозиторію проекту, має історію змін, версії та відповідальних авторів — так само як і будь-який інший файл коду.

2. Що таке Single Source of Truth?
Single Source of Truth (SSoT) — це принцип, за яким кожен факт або вимога описується рівно в одному місці, а всі інші документи лише посилаються на нього. У нашому проекті SSoT для функціональних вимог є SSD.md — саме там описані всі FR-01..FR-07. Test Strategy та Traceability Matrix не дублюють вимоги, а посилаються на їх ідентифікатори з SSD. Це виключає ситуацію, коли в різних документах одна й та ж вимога описана по-різному.

3. Чому Markdown є базовим форматом для документації?
Markdown обраний з кількох причин. По-перше, це простий текстовий формат — його можна читати навіть без рендерингу, на відміну від .docx або .pdf. По-друге, він чудово інтегрується з Git: зміни у Markdown-файлах видно у звичайному git diff, що дозволяє робити нормальний code review документації. По-третє, з Markdown легко генерувати статичні сайти (MkDocs, Docusaurus) або конвертувати у PDF та HTML. Нарешті, він підтримується GitHub, GitLab та більшістю інструментів розробки "з коробки".

4. Як забезпечується актуальність документації у DaC-підході?
Актуальність забезпечується організаційно і технічно одночасно. Організаційно — діє правило: зміна в коді або вимогах без оновлення відповідного документа не може бути змержена (це перевіряється на code review). Технічно — CI/CD pipeline автоматично перегенеровує та публікує сайт документації при кожному merge у main, тому стара версія сайту фізично замінюється новою. Додатково версійність через Git теги (v1.0, v1.1) дозволяє завжди знайти документацію, що відповідає конкретному релізу системи.
